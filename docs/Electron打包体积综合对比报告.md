# Electron Forge 与 Electron Builder 打包方案体积对比报告

> **测试时间**: 2026-02-24  
> **测试环境**: macOS (arm64), v1.1.1  
> **背景说明**: 鉴于项目在使用 Electron 默认构建工具打包时出现了应用体积过大的问题，为了探寻更优的体积压缩和优化策略，我们针对主流的构建方案展开了深度测试，旨在为本项目敲定一套既能兼顾开发体验又能最小化分发体积的最佳打包引擎选型。

---

## 1. 体积差异综合对比

评价打包体积通常需要关注两个重点维度：
1. **压缩后的安装包体积**（如 `.zip`, `.dmg`）：直接影响用户的拉取、下载速度和体验。
2. **解压后的应用体积**（即实际占用的 `.app` 大小）：影响用户本地磁盘空间的占用。

### 1.1 压缩后体积（用户下载大小）

| 打包工具 | 分发格式 | 文件大小 | 结果评价 |
| -------- | -------- | -------- | -------- |
| **Electron Builder** | `.zip` | **212MB** | 🏆 **显著优于 Forge** |
| **Electron Builder** | `.dmg` | **208MB** | 表现优良 |
| **Electron Forge** | `.zip` | **321MB** | 存在负压缩问题 |

**🔍 深度解析**：
Builder 的压缩算法更优秀且配置了“最大压缩”，其生成的最终下载文件比 Forge 减小了约 **109MB（优化幅度约 34%）**。值得注意的是，Forge 使用的是标准 ZIP 压缩，由于不适配某些文件结构，导致压缩后的体积反而大于原生释放体积（250MB）。

### 1.2 解压后体积（磁盘占用大小）

| 打包工具 | 未压缩 `.app` 大小 | 结果评价 |
| -------- | ------------------ | -------- |
| **Electron Forge** | **250MB** | 🏆 **显著优于 Builder** |
| **Electron Builder** | **514MB** | 包含严重依赖冗余 |

**🔍 深度解析**：
Builder 在解压体积上完败，比 Forge 大了足足 **264MB**。
其**根本原因**在于 Electron Builder 的默认打包机制：
1. 自动无视 Vite 的 bundle 产物（在 Vite 中 `.vite/build/main.js` 已打包所有必要运行时代码，从 29KB 变为 218KB）。
2. 即使前端依赖已打包，Builder 依然会将 `package.json` 中的 `dependencies` 全部塞入生成的 `app.asar` 内的 `node_modules` 中。
3. `files` 配置中的 exclude 排除规则默认对依赖项无效。

---

## 2. 优缺点全面总结

### 📦 Electron Builder 阵营

**优点 (Pros):**
1. **极致的下载体积**：高度优化的压缩比（ZIP 文件 212MB），显著降低新用户的入门等待时间。
2. **完整的原生分发格式**：完美支持 macOS 原生 DMG 镜像拖拽安装包的创建。
3. **强大的 OTA 增量更新**：内置 Blockmap 支持，对应用未来的自动增量更新极度友好。
4. **高度可定制 API**：提供了极高自定义自由度的配置文件。

**缺点 (Cons):**
1. **严重的磁盘空间冗余**：生搬硬套打包了无用的 `node_modules`，导致本地解压后占用多达 500MB 以上。解决该问题需要编写复杂的 `afterPack` 脚本强行介入 Asar 清理。
2. **构建链路略重且慢**：各种深度合并和最高级算法大大拉长了流水线的打包耗时。
3. **配置学习成本高**。

### ⚡️ Electron Forge 阵营

**优点 (Pros):**
1. **本分克制的应用体积**：打包极简纯粹，`app.asar` 仅有 2.1MB，解压后整体体积控制在十分合理的 250MB，无垃圾依赖。
2. **非凡的打包速度**：没有繁重的后处理流程，非常适合快节奏的修改与验证。
3. **开箱即用的生态融合**：与 Vite 等前端官方构建插件体系融合得非常丝滑。

**缺点 (Cons):**
1. **算法薄弱导致分发体积暴涨**：令人头疼的负压缩，分发包高达 320MB+。
2. **缺乏高级分发与更新能力**：默认不提供极具质感的 DMG 生成以及后续平滑的增量更新机制。

---

## 3. 面向本项目的最终选型建议

在了解了它们截然不同的“性格”后，我们目前的工程可以采用以下策略进行应对：

### 🎯 场景 A：开发调试与日常验证
**优先使用：Electron Forge (`npm start`)**
- 原因：快速构建，所见即所得。产物体积实在，不需要浪费时间进行深度压缩和格式转换。

### 🎯 场景 B：生产级别的线上正式发布
**优先使用：Electron Builder (`npm run build:builder:mac`)**
- 原因：作为一个成熟的商业 / C端 面向桌面应用，**用户的“首屏安装下载体验”（DMG 和小巧的体积）以及“日后静默平滑增量升级”是重中之重**。
- 尽管用户在安装到 `/Applications` 目录后会多占用 260MB 多一点磁盘空间，但在动辄数百 G 到几 T 固态硬盘的今天，绝大部分用户对这二三百兆根本**不敏感**。如果非要极致追求解压体积，则可以考虑尝试通过 `afterPack` 钩子或临时移动依赖项到 `devDependencies` 的方式擦除多余 `node_modules`。

---

**最终定论：**
短期来看，开发链路保留 Forge 作为日常引擎，而需要面对真实用户公测下发时，启用 Builder 配合 DMG 和增量更新补丁作为主力打法。
